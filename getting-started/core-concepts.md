# Core concepts

This page defines the vocabulary used across the UI, CLI, and API. If you've read the introduction and want to know what a "baseline" or a "job execution" is before touching the product, start here.

## Provider

A **provider** is a connected cloud account. Each provider record stores encrypted credentials, a display name, a sync status, and the provider type (`aws`, `gcp`, `azure`, or `kubernetes`). One InfraAudit account can connect many providers at once — for example, three AWS accounts, one GCP project, and two Kubernetes clusters.

Providers are created through `POST /api/v1/providers/{provider}/connect` or from the CLI with `infraudit provider connect aws`. Disconnecting a provider removes its credentials but keeps the historical scan data.

See also: [Connecting cloud accounts](../platform/connecting-cloud-accounts.md), [Integrations: AWS](../integrations/aws.md).

## Resource

A **resource** is a single item discovered by a provider sync: an EC2 instance, an S3 bucket, a GCS bucket, an Azure VM, a Kubernetes deployment. Every resource has:

- A `resource_id` (InfraAudit's internal ID) and an `external_id` (the cloud provider's ARN, URI, or name)
- A `provider_id` pointing to the cloud account it belongs to
- A `resource_type` (e.g. `ec2_instance`, `s3_bucket`, `rds_instance`)
- A `region`
- A `status` (active, stopped, deleted)
- Attached metadata: tags, configuration snapshot, cost data

Resources are the unit that everything else hangs off: drifts, baselines, vulnerabilities, recommendations, and cost allocations all reference a `resource_id`.

## Baseline

A **baseline** is a captured snapshot of a resource's configuration at a point in time. Once a baseline exists, drift detection can compare the current state against it and flag differences.

Baselines are created manually (you pick the "known good" state) or automatically after initial discovery. A single resource can have multiple baselines across time; the most recent one is the default comparison target.

See also: [Concepts: Baselines](../concepts/baselines.md).

## Drift

A **drift** is a detected difference between a resource's current configuration and its baseline. Drifts have three type categories:

- **Configuration drift** — something changed that was not expected (e.g. a security group rule was added)
- **Security drift** — a change that weakens the security posture (e.g. a bucket became public)
- **Compliance drift** — a change that violates a compliance control (e.g. encryption was disabled)

Each drift carries a severity (`critical`, `high`, `medium`, `low`) and a status (`detected`, `investigating`, `resolved`). The drift scanner runs on a cron schedule (default: every 4 hours) and can be triggered manually.

See also: [Concepts: Drift detection](../concepts/drift-detection.md).

## IaC definition

An **IaC definition** is an Infrastructure-as-Code file uploaded to InfraAudit: a Terraform `.tf` file, a CloudFormation YAML template, or a Kubernetes manifest. Once uploaded, InfraAudit parses the file, identifies the resources it declares, and can compare them against live resources to detect IaC drift: cases where the live state no longer matches the declared state.

See also: [Integrations: Terraform](../integrations/terraform.md).

## Vulnerability

A **vulnerability** is a CVE matched against a resource, usually a container image or a runtime package. Vulnerabilities are produced by the Trivy scanner, enriched with NVD metadata, and stored with a CVSS severity score. Each vulnerability has:

- A CVE ID
- A severity (`critical`, `high`, `medium`, `low`)
- A status (`open`, `fixed`, `ignored`)
- The affected resource(s) and package version
- A fix version, if one exists

## Alert

An **alert** is a user-facing notification about something that needs attention. Alerts are generated automatically from drifts, vulnerabilities, cost anomalies, and failed compliance controls, or created manually. Each alert has a severity, a status (`open`, `acknowledged`, `resolved`), and a type (`security`, `compliance`, `performance`, `cost`).

Alerts are delivered to the channels configured under notification preferences: Slack, email, and custom webhooks.

## Recommendation

A **recommendation** is an AI-generated or rule-based suggestion about how to fix a finding or save money. Recommendations have three types:

- **Cost** — right-sizing, Reserved Instance purchase, Spot migration, idle resource cleanup
- **Security** — hardening changes based on drifts and vulnerabilities
- **Performance** — resource scaling or configuration adjustments

Each recommendation carries an estimated impact: monthly savings for cost, risk reduction for security. Recommendations have a status (`pending`, `applied`, `dismissed`) and are generated by Google Gemini when `GEMINI_API_KEY` is set, or by a rule-based fallback otherwise.

See also: [Concepts: AI recommendations](../concepts/ai-recommendations.md).

## Assessment (compliance)

A **compliance assessment** is a single run of a compliance framework against your connected resources. The framework engine supports CIS Benchmarks, SOC2, NIST 800-53, PCI-DSS, and HIPAA. Each assessment produces:

- A list of controls evaluated
- A pass/fail status per control
- A mapping from failed controls to the resources that caused the failure
- An overall score

Assessments can be exported to PDF or CSV for audit evidence.

See also: [Concepts: Compliance frameworks](../concepts/compliance-frameworks.md).

## Job and job execution

A **job** is a scheduled unit of background work defined by a cron expression. Job types include:

- `resource_sync` — pull the latest inventory from a provider
- `drift_detection` — run the drift scanner across all providers
- `vulnerability_scan` — run Trivy against container images
- `cost_sync` — pull billing data from cloud providers
- `compliance_check` — run one or more compliance frameworks

Each time a job runs, InfraAudit records a **job execution** with a start time, end time, status (`running`, `succeeded`, `failed`), and a log snippet. Default schedules live in `.env.example` and can be overridden per job.

See also: [Concepts: Job scheduler](../concepts/job-scheduler.md).

## Remediation action

A **remediation action** is a proposed or approved fix that InfraAudit can apply on your behalf. Remediations are generated from drifts and vulnerabilities, go through an optional approval step, and then execute against the cloud provider API. Every action has a rollback path so you can revert if the fix causes a regression.

Remediation actions have a status: `suggested`, `pending_approval`, `approved`, `executing`, `completed`, `failed`, or `rolled_back`.

See also: [Concepts: Remediation workflow](../concepts/remediation-workflow.md).

## Webhook

A **webhook** is an outbound HTTP endpoint that InfraAudit POSTs to when a subscribed event fires. Events include `drift.detected`, `cost.anomaly`, `compliance.violation`, `alert.created`, and more. Each webhook is signed with an HMAC-SHA256 signature using a per-webhook secret so the receiver can verify authenticity.

See also: [API: Webhooks](../api/webhooks/README.md).

## Notification preferences

**Notification preferences** are per-user, per-channel settings that control which events go to which delivery channel. A user can enable Slack for critical alerts only, route all cost anomalies to email, and fire webhooks for everything else.

## User, team, and API key

- A **user** is an authenticated identity, backed by Supabase Auth. Users have a role (`user`, `admin`, `support`) and a plan type (`free`, `starter`, `pro`, `enterprise`).
- A **team** is a group of users who share providers, resources, and findings. Team members are managed through the Settings → Team API.
- An **API key** is a long-lived credential (Bearer token) a user can create for scripting and integrations. API keys authenticate the same endpoints as JWT tokens.
